# 原理与试错路径

## 最终生效路径

最终可用方案是 DLSSTweaks `winmm.dll` wrapper：

```text
<Win64>\winmm.dll
<Win64>\dlsstweaks.ini
```

这里的 `winmm.dll` 来自 DLSSTweaks release 的 `dxgi.dll`，安装时复制并改名。

异环的 Win64 目录是：

```text
Client\WindowsNoEditor\HT\Binaries\Win64
```

目标程序是：

```text
HTGame.exe
```

## 为什么不用 dxgi.dll

`dxgi.dll` 是 DLSSTweaks 常见入口，但异环会检测到它。被检测后，游戏不能稳定进入可用状态。

因此最终避开 `dxgi.dll` 文件名，改用 `winmm.dll`。

## 为什么 winmm.dll 可用

异环运行链中会加载依赖 WinMM 的组件。`winmm.dll` 不是最终 DLSS DLL，而是一个早期加载入口。DLSSTweaks 通过这个入口进入进程后，再 hook DLSS 参数。

成功日志示例：

```text
DLSSTweaks v0.310.2.1, by emoose: WINMM.dll wrapper loaded
Config read from ... dlsstweaks.ini
DLSS functions found & parameter hooks applied
```

## 试过但不作为最终方案

### NVIDIA Profile 全局比例

DLSSTweaks ConfigTool 可以写 NVIDIA DRS 全局 DLSS 比例，例如 `GlobalForcedScale`。

这个方案不放 DLL，适合不能注入 DLL 的游戏，但实测低于 `0.333` 时会被限制到约 720p，无法达到 648p/680p 目标。

本项目不会写 `GlobalForcedScale`。

### nvngx.dll

理论上 `nvngx.dll` 方案兼容性好，但本游戏没有从 `HTGame.exe` 同目录加载这个本地 DLL，未生成 DLSSTweaks 日志。

### XInput1_4.dll

`XInput1_4.dll` 看起来可能被导入，但在本游戏中属于延迟加载路径，实际没有稳定触发，所以不作为最终方案。

### UE Engine.ini / ConsoleVariables.ini

试过 `r.ScreenPercentage`、`r.DynamicRes.*`、`Rendering.AutoScreenPercentage` 等 UE 配置。游戏会覆盖或忽略这些 loose config，不能改变 DLSS HUD 中的实际内部渲染分辨率。

这些配置不属于最终方案，发布版不写它们。

## 当前方案的边界

- 只对选中的游戏目录生效。
- 需要游戏重启后读取。
- 不改变 NVIDIA App 全局比例。
- HUD 开关是全局注册表，只用于显示调试信息。
